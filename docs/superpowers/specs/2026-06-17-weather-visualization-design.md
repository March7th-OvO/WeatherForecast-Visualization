# 基于 Python 的天气预报可视化分析系统设计文档

## 1. 项目概述

### 1.1 项目名称
基于 Python 的天气预报可视化分析系统

### 1.2 项目目标
围绕课程设计需求，构建一个可在 Windows 笔记本本机开发和运行的天气数据采集、清洗、存储、分析与网页可视化系统。系统以中国天气网为主要数据来源，完成不少于 2 万条天气数据的采集，并通过 Flask 和 ECharts 提供可视化展示。

### 1.3 项目定位
- 本机真实可运行
- 适合课程展示与答辩
- 大数据扩展链路在设计层体现，但不强依赖本机部署

### 1.4 数据目标
项目目标总量为不少于 `20000 条` 天气数据。

最终实现时，基于中国天气网当前稳定可抓取的 15 天天气页面，采用“扩大城市/站点数量”的方式完成数据量目标。实际验证口径为约 `1370 个城市/站点 × 15 天`，清洗后成功入库 `20550 条` 数据。

## 2. 范围与约束

### 2.1 本次实现范围
- Python 爬取中国天气网天气数据
- 原始数据保存为 CSV 文件
- 对 CSV 数据进行清洗与转换
- 将清洗后的结构化数据写入 MySQL
- 使用 Flask 提供网页服务
- 使用 ECharts 实现 4 个页面的可视化展示

### 2.2 页面范围
- 首页总览
- 天气地图
- 天气分析
- 历史天气查询

### 2.3 非目标
- 本次不强制在 Windows 本机真实部署 Kafka、Flume、HDFS、Hive、HBase、Spark
- 不追求复杂用户权限、登录、后台管理等功能
- 不做高并发或分布式生产级部署

## 3. 总体架构

### 3.1 主链路
`中国天气网爬虫 -> 原始天气 CSV -> Python 清洗转换 -> MySQL -> Flask -> ECharts 可视化页面`

### 3.2 扩展链路
为满足课程“大数据流程”表达，系统设计中补充如下扩展架构：

`爬虫/采集程序 -> Kafka/Flume -> HDFS/Hive -> Spark -> MySQL/可视化系统`

说明：
- 主链路是真实实现并可运行的部分
- 扩展链路在文档、流程图、目录和预留脚本入口中体现
- 这样既控制了 Windows 本机开发风险，也保留了完整的大数据架构表达

## 4. 系统模块设计

### 4.1 数据采集模块
职责：
- 读取目标城市列表
- 按城市和日期范围请求天气页面
- 解析天气信息
- 将结果写入原始 CSV

采集策略：
- 以城市为单位分批抓取
- 采集过程中记录来源链接和抓取日志
- 单条失败时跳过并记录，不影响整批采集

### 4.2 原始数据存储模块
职责：
- 保存爬虫直接输出的原始数据
- 为后续数据核验、去重、追溯提供依据

存储形式：
- CSV 文件按批次或按城市保存
- 文件命名带时间戳，便于区分采集批次

### 4.3 数据清洗与转换模块
职责：
- 统一字段格式
- 清理无效字符
- 提取温度数值
- 去重
- 输出可入库的数据

核心清洗规则：
- 城市名标准化
- 日期格式统一为标准日期格式
- 天气类型去除多余空格和异常符号
- 最高温、最低温转换为数值类型
- 对重复记录按城市和日期去重

### 4.4 数据存储模块
职责：
- 管理 MySQL 数据表
- 执行批量入库
- 为 Flask 页面查询提供结构化数据基础

### 4.5 分析与接口模块
职责：
- 对天气数据做统计分析
- 向前端页面返回概览、趋势、分类统计与明细数据
- 统一封装查询逻辑，避免模板层直接拼装复杂 SQL

### 4.6 可视化展示模块
职责：
- 展示课程答辩需要的图表和页面
- 提供直观的数据总览、地图展示、趋势分析和历史查询功能

## 5. 数据设计

### 5.1 原始 CSV 字段
本次字段数量控制在课程展示友好的范围内，原始和入库核心字段保持一致，避免无谓复杂化。

核心字段如下：
- `id`
- `city_name`
- `weather_date`
- `weather_type`
- `high_temp`
- `low_temp`
- `wind_level`

说明：
- 若后续采集稳定，可补充来源链接和采集时间到原始文件中
- 但核心可视化与分析逻辑以这 7 个字段为基础

### 5.2 MySQL 主表
建议主表名：`weather_daily`

字段定义：
- `id`：主键，自增
- `city_name`：城市名
- `weather_date`：日期
- `weather_type`：天气状况，如晴、阴、小雨
- `high_temp`：最高温
- `low_temp`：最低温
- `wind_level`：风力等级

### 5.3 数据表设计原则
- 字段精简，便于采集和答辩讲解
- 支撑 4 个页面的全部需求
- 以一张主表为核心，必要统计可在查询时聚合，避免早期过度建模

## 6. 页面功能设计

### 6.1 首页总览
展示内容：
- 天气记录总数
- 已采集城市数
- 全部数据中的最高温
- 全部数据中的最低温
- 近 30 天平均最高温趋势折线图
- 天气类型分布柱状图

页面风格：
- 左侧深色导航栏
- 右侧卡片 + 图表布局
- 视觉效果贴近用户提供的后台页面参考图

### 6.2 天气地图
展示内容：
- 全国城市温度分布地图
- 支持查看平均最高温或平均最低温

实现要点：
- 采用中国地图可视化
- 城市与统计值绑定后进行着色展示

### 6.3 天气分析
展示内容：
- 城市平均最高温 Top10 条形图
- 城市平均最低温 Top10 条形图
- 天气类型占比饼图
- 某城市近 30 天温度变化折线图

目标：
- 集中呈现系统的主要统计结论
- 支撑课程答辩中的“分析、转化、演示”环节

### 6.4 历史天气查询
展示内容：
- 按城市名查询历史天气
- 表格展示指定城市的天气明细记录

字段展示：
- 城市名
- 日期
- 天气状况
- 最高温
- 最低温
- 风力等级

## 7. 数据处理流程

### 7.1 采集流程
1. 读取目标城市列表
2. 拼接请求地址
3. 抓取天气页面
4. 解析页面内容
5. 写入原始 CSV

### 7.2 清洗流程
1. 读取原始 CSV
2. 标准化字段
3. 清理空值与异常值
4. 提取温度数值
5. 去重
6. 输出清洗结果

### 7.3 入库流程
1. 读取清洗后的数据
2. 检查表结构
3. 按批次写入 MySQL
4. 记录导入结果和错误日志

### 7.4 展示流程
1. Flask 路由接收页面请求
2. 后端查询 MySQL
3. 返回模板或 JSON 数据
4. ECharts 渲染图表

## 8. 异常处理设计

### 8.1 爬虫异常
- 请求超时或失败时进行记录
- 对单页失败采用跳过策略，避免整批任务中断
- 对城市级失败提供重试入口

### 8.2 数据异常
- 空值统一清洗
- 温度无法转换时记录异常并剔除
- 重复数据通过城市名 + 日期维度去重

### 8.3 入库异常
- 批量插入失败时输出失败原因
- 保留失败批次文件，方便重新导入

### 8.4 页面异常
- 查询为空时返回“暂无数据”
- 图表无数据时展示空状态
- 搜索不存在的城市时给出明确提示

## 9. 测试与验证设计

### 9.1 采集验证
- 抽样检查 CSV 是否包含目标字段
- 检查数据总量是否达到目标级别
- 核对不同城市和不同日期是否真实分布

### 9.2 清洗验证
- 检查温度字段是否为数值
- 检查日期格式是否统一
- 检查是否存在重复数据

### 9.3 入库验证
- 校验 MySQL 记录数
- 抽样核对 CSV 与数据库内容是否一致

### 9.4 页面验证
- 首页卡片与数据库统计结果一致
- 地图可正常渲染
- 分析图表可正常显示
- 历史查询可返回正确记录

## 12. 风险与取舍

### 12.1 主要风险
- 中国天气网历史数据可抓范围可能有限
- 抓取频率过高时存在反爬风险
- 2 万条数据采集耗时可能较长

### 12.2 对应策略
- 采用多城市分批抓取
- 加入请求间隔与失败重试
- 先完成小样本通路，再扩大到 2 万条

### 12.3 取舍说明
本次优先保证：
- 系统真实可运行
- 页面完整可演示
- 采集、清洗、入库、分析链路清晰

不优先追求：
- 全量大数据组件本机部署
- 复杂企业级系统能力

## 13. 结论

本项目最终方案为：

`中国天气网爬虫 + CSV 原始数据 + Python 清洗 + MySQL + Flask + ECharts`

在 Windows 笔记本上完成开发和运行，围绕 4 个页面实现天气数据的可视化分析展示，并以 `100 个城市 × 200 天` 作为数据目标。扩展层通过架构设计体现 Kafka、Flume、HDFS、Hive、Spark 的位置，为课程汇报保留完整的大数据系统表达。
