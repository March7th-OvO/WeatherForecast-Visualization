# Flask 项目在 Ubuntu 26.04 与 CentOS 7 两种虚拟机上的运行复盘

## 1. 背景

今天的目标是把一个 Flask 项目 `xhs-WeatherForecastSystem` 放到虚拟机中运行，并能从宿主机浏览器访问。

项目本身是一个 Python Web 项目，使用 Flask 启动，同时依赖 `pandas`、`numpy` 等 Python 包。因此整个过程不仅涉及文件上传和 SSH 连接，还涉及 Python 版本、虚拟环境、依赖安装、系统编译环境等问题。

这次一共涉及两台虚拟机：

1. Ubuntu 26.04 LTS 虚拟机
2. CentOS 7 虚拟机

两台机器的问题完全不同：

- Ubuntu 26.04 的问题是系统太新，默认 Python 版本过高，导致老项目依赖不兼容。
- CentOS 7 的问题是系统太老，默认 Python 只有 2.7.5，GCC 也很旧，导致新版本科学计算包无法编译。

最终形成的经验是：**不要直接使用系统默认 Python 跑项目，应该为项目创建独立的 Python 虚拟环境，并根据系统环境选择合适的 Python 和依赖版本。**

## 4. Ubuntu 中遇到的第一个问题：`python` 命令不存在

在项目目录里执行：

```bash
python -m venv venv
```

结果提示：

```text
Command 'python' not found, did you mean:
  command 'python3' from deb python3
  command 'python' from deb python-is-python3
```

原因是 Ubuntu 中默认命令通常叫 `python3`，不一定提供 `python` 这个命令。

因此应该使用：

```bash
python3 --version
python3 -m venv venv
```

如果缺少 venv 模块，需要安装：

```bash
sudo apt update
sudo apt install python3-venv python3-pip -y
```

然后重新创建虚拟环境：

```bash
python3 -m venv venv
source venv/bin/activate
```

激活成功后，命令行前面会出现：

```bash
(venv)
```

## 5. Ubuntu 中遇到的第二个问题：pandas 在 Python 3.14 下编译失败

安装依赖时出现：

```text
error: metadata-generation-failed
```

进一步查看错误，发现是 `pandas` 编译失败：

```text
aggregations.cpython-314-x86_64-linux-gnu.so
/usr/include/python3.14
```

这说明当前环境使用的是 Python 3.14。

继续往下看，错误出现在 pandas 的 C/C++ 扩展编译阶段：

```text
error: standard attributes in middle of decl-specifiers
```

这个问题的核心原因是：

```text
Ubuntu 26.04 默认 Python 太新，项目依赖中的 pandas 版本不适配 Python 3.14。
```

对于普通 Flask 老项目，不建议直接使用 Python 3.14。更稳妥的是使用 Python 3.12 或 Python 3.11。

------

## 6. Ubuntu 中解决 Python 版本问题：使用 uv 安装 Python 3.12

尝试使用 apt 安装 Python 3.12：

```bash
sudo apt install python3.12 python3.12-venv python3.12-dev -y
```

结果失败：

```text
Unable to locate package python3.12
```

说明 Ubuntu 26.04 的系统源里没有直接提供这个版本。

于是改用 `uv` 安装独立 Python 版本。

先安装 curl：

```bash
sudo apt update
sudo apt install curl -y
```

安装 uv：

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

让当前终端识别 uv：

```bash
export PATH="$HOME/.local/bin:$PATH"
```

进入项目目录，删除旧虚拟环境：

```bash
cd ~/xhs-WeatherForecastSystem
deactivate 2>/dev/null
rm -rf venv
```

用 Python 3.12 创建虚拟环境：

```bash
uv venv --python 3.12 venv
source venv/bin/activate
python --version
```

最终显示：

```text
Python 3.12.13
```

这说明 Python 3.12 虚拟环境创建成功。

## 7. Ubuntu 中遇到的第三个问题：虚拟环境没有 pip

进入 Python 3.12 虚拟环境后，执行：

```bash
python -m pip install --upgrade pip setuptools wheel
```

结果报错：

```text
No module named pip
```

随后直接执行：

```bash
pip install -r requirements.txt
```

又触发了 Ubuntu 的系统保护机制：

```text
error: externally-managed-environment
```

原因是当前虚拟环境里没有 pip，而裸用 `pip` 时调用到了系统 pip。

解决办法是直接使用 `uv pip` 安装依赖：

```bash
uv pip install pip setuptools wheel -i https://pypi.tuna.tsinghua.edu.cn/simple
uv pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
```

这样就避免了系统 pip 和虚拟环境 pip 混用的问题。

## 8. Ubuntu 中运行 Flask 后无法访问的问题

依赖安装完成后，运行：

```bash
flask --app app run
```

终端显示：

```text
Running on http://127.0.0.1:5000
```

但是在 Windows 浏览器访问：

```text
http://127.0.0.1:5000
```

看不见页面。

原因是：

```text
虚拟机里的 127.0.0.1 只代表虚拟机自己，不代表 Windows 宿主机。
```

如果 Flask 只监听 `127.0.0.1`，宿主机无法通过虚拟机 IP 访问。

正确启动方式是：

```bash
flask --app app run --host=0.0.0.0 --port=5000
```

或者：

```bash
python -m flask --app app run --host=0.0.0.0 --port=5000
```

然后在 Windows 浏览器访问：

```text
http://192.168.17.129:5000
```

如果仍然打不开，需要检查防火墙：

```bash
sudo ufw status
```

如果防火墙启用，则放行 5000 端口：

```bash
sudo ufw allow 5000
```

## 12. CentOS 7 上传项目时的路径问题

在 Windows 中执行了类似命令：

```powershell
scp -r E:\MyProjects\xhs-WeatherForecastSystem root@192.168.17.128:/home/root
```

一开始以为项目会出现在：

```bash
/home/root/xhs-WeatherForecastSystem
```

但在 CentOS 中查找：

```bash
find / -type d -name "xhs-WeatherForecastSystem" 2>/dev/null
```

没有找到。

查看 `/home/root`：

```bash
ls /home/root
```

结果显示：

```text
app  data  docs  README.md  requirements.txt  scripts  spider  sql  static  templates  tests
```

这说明项目内容已经上传成功，只是没有保留最外层的 `xhs-WeatherForecastSystem` 文件夹名。

也就是说，项目根目录就是：

```bash
/home/root
```

而不是：

```bash
/home/root/xhs-WeatherForecastSystem
```

如果想整理成正常结构，可以新建目录并移动：

```bash
mkdir -p /root/xhs-WeatherForecastSystem
mv /home/root/* /root/xhs-WeatherForecastSystem/
mv /home/root/.* /root/xhs-WeatherForecastSystem/ 2>/dev/null
```

之后进入：

```bash
cd /root/xhs-WeatherForecastSystem
```

## 13. CentOS 7 上遇到的 numpy 编译失败问题

在 CentOS 7 中使用 uv 创建虚拟环境后，安装依赖时出现：

```text
Failed to build numpy==2.5.0
```

关键错误是：

```text
C compiler for the host machine: cc (gcc 4.8.5)
ERROR: Problem encountered: NumPy requires GCC >= 9.3
```

这说明 CentOS 7 的 GCC 版本太老：

```text
GCC 4.8.5
```

而新版本 NumPy 需要：

```text
GCC >= 9.3
```

换句话说，问题不是 Flask，也不是 uv，而是：

```text
CentOS 7 太老，新版本 numpy 没有合适的二进制 wheel，于是退回源码编译，而源码编译需要更高版本 GCC。
```

## 15. CentOS 7 上的核心解决思路：约束 numpy 版本

由于 `pandas==2.2.2` 会依赖 numpy，但它没有强制限制 numpy 必须小于 2，因此 uv 自动选择了较新的 numpy，例如：

```text
numpy==2.4.6
```

这会导致 CentOS 7 编译失败。

所以需要使用 constraints 文件强制约束版本。

创建约束文件：

```bash
cat > constraints.txt <<'EOF'
numpy==1.26.4
pandas==2.2.2
EOF
```

然后安装依赖时加上 `-c constraints.txt`：

```bash
uv pip install -r requirements.txt -c constraints.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
```

为了避免源码编译，可以进一步加：

```bash
uv pip install -r requirements.txt -c constraints.txt --only-binary=:all: -i https://pypi.tuna.tsinghua.edu.cn/simple
```

这条命令的含义是：

```text
只允许安装预编译 wheel，不允许源码编译。
```

如果没有合适 wheel，它会直接失败，而不是尝试用 CentOS 7 的老 GCC 编译。

也可以先单独安装核心依赖：

```bash
uv pip install "numpy==1.26.4" "pandas==2.2.2" --only-binary=:all: -i https://pypi.tuna.tsinghua.edu.cn/simple
```

然后再安装项目依赖：

```bash
uv pip install -r requirements.txt -c constraints.txt --only-binary=:all: -i https://pypi.tuna.tsinghua.edu.cn/simple
```

## 18. 本次排障得到的结论

这次部署最大的收获不是某一条命令，而是环境管理思路。

第一，不要随便替换系统 Python。Ubuntu、CentOS 都有自己的系统 Python 依赖，尤其是 CentOS 7，`yum` 依赖 Python 2.7，不能直接把 `/usr/bin/python` 改成 Python 3。

第二，项目应该使用独立虚拟环境。不同项目需要不同 Python 版本和依赖版本，使用 `venv` 或 `uv venv` 可以把项目环境和系统环境隔离开。

第三，系统太新和太旧都会出问题。Ubuntu 26.04 的问题是 Python 太新，导致旧依赖不兼容；CentOS 7 的问题是系统太旧，GCC 太旧，导致新依赖无法编译。

第四，遇到 numpy、pandas 这类科学计算包时，要优先使用 wheel，不要轻易源码编译。源码编译对 GCC、glibc、Python 头文件、BLAS 等系统依赖要求很高，尤其不适合 CentOS 7 这种老系统。

第五，Flask 在虚拟机里运行时，要监听 `0.0.0.0`，否则只能在虚拟机内部访问：

```bash
python -m flask --app app run --host=0.0.0.0 --port=5000
```

第六，上传文件后不要凭感觉猜目录，应该用：

```bash
pwd
ls
find / -type d -name "项目名" 2>/dev/null
```

确认真实路径。

最终，这个项目在两种虚拟机上的运行问题，本质上都是 Python 工程环境兼容性问题。解决方式不是暴力升级系统，而是通过 `uv`、虚拟环境和依赖版本约束，给项目构造一个合适的运行环境。⚙️
